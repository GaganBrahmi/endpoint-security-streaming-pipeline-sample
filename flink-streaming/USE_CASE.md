# Use Case: Endpoint Security Event Processing

## Overview

This application processes endpoint security events in real-time to provide comprehensive threat detection, attack pattern analysis, and compliance reporting for enterprise security operations centers (SOCs).

## Business Context

### Problem Statement

Modern enterprises face increasing cybersecurity threats targeting endpoints (laptops, desktops, servers, mobile devices). Security teams need to:

1. **Detect threats in real-time** - Identify malicious activity as it happens
2. **Analyze attack patterns** - Understand coordinated attack campaigns
3. **Generate compliance reports** - Meet regulatory requirements (SOC 2, PCI-DSS, HIPAA)
4. **Investigate incidents** - Query historical data for forensic analysis
5. **Monitor security posture** - Track metrics across time windows

### Solution

This PyFlink streaming application provides:

- **Real-time threat detection** using sliding windows for continuous monitoring
- **Attack campaign identification** using session windows to group related events
- **Compliance reporting** using tumbling windows for fixed-time period summaries
- **Historical analysis** via Iceberg tables queryable in Amazon Athena
- **Scalable processing** handling millions of events per day

## Data Flow

### 1. Event Generation

Events are generated by endpoint security agents installed on devices:

```
Endpoint Device (Laptop/Server/Mobile)
  ↓
Security Agent (monitors file, process, network activity)
  ↓
Event Detection (threat analysis, policy enforcement)
  ↓
JSON Event Creation
  ↓
Kafka Producer → Amazon MSK Topic: endpoint_logs
```

### 2. Stream Processing

PyFlink application consumes and processes events:

```
Kafka Source (endpoint_logs topic)
  ↓
Event Parsing (JSON to Table API)
  ↓
Watermark Assignment (event_time - 5 seconds)
  ↓
┌─────────────────────────────────────────┐
│ Parallel Processing Streams             │
├─────────────────────────────────────────┤
│ 1. Raw Events → endpoint_data_flink     │
│ 2. Tumbling Windows → 4 tables          │
│ 3. Sliding Windows → 1 table            │
│ 4. Session Windows → 1 table            │
└─────────────────────────────────────────┘
  ↓
Iceberg Sink (7 tables total)
```

### 3. Data Storage

Processed data is stored in Iceberg tables:

```
Iceberg Tables (AWS Glue Catalog)
  ↓
Parquet Files (S3 Warehouse)
  ↓
Partitioned by Date (year/month/day)
  ↓
Queryable via Amazon Athena
```

## Windowing Strategies

### Why Multiple Window Types?

Different security use cases require different time perspectives:

| Window Type | Duration | Use Case | Update Frequency |
|-------------|----------|----------|------------------|
| **Session** | Variable (5 min gap) | Attack campaign detection | When session ends |
| **Sliding** | 5 min / 1 min slide | Real-time alerting | Every minute |
| **Tumbling 5min** | 5 minutes | Operational monitoring | Every 5 minutes |
| **Tumbling 15min** | 15 minutes | Tactical analysis | Every 15 minutes |
| **Tumbling 30min** | 30 minutes | Hourly briefings | Every 30 minutes |
| **Tumbling 60min** | 60 minutes | Daily reports | Every hour |

### 1. Session Windows (Attack Campaign Detection)

**Purpose**: Group related security events that occur in bursts

**Configuration**: 5-minute gap
- Events within 5 minutes of each other are grouped
- Session ends after 5 minutes of inactivity

**Use Case**: Detect coordinated attacks
```
Example: Ransomware Attack
10:00:00 - Initial malware execution
10:00:15 - File encryption starts
10:01:30 - Network beacon to C&C server
10:02:45 - Lateral movement attempt
10:03:10 - Additional file encryption
[5 minute gap - no activity]
10:08:15 - Session closes, alert generated

Result: All 5 events grouped as single attack campaign
```

**Table**: `threat_sessions_5min_gap`

**Metrics**:
- Session duration (how long attack lasted)
- Event burst detection (rapid succession of events)
- Total events in session
- Threat concentration

### 2. Sliding Windows (Real-Time Alerting)

**Purpose**: Continuous monitoring with overlapping windows

**Configuration**: 5-minute window, 1-minute slide
- New window every minute
- Each window covers last 5 minutes
- Windows overlap by 4 minutes

**Use Case**: Real-time threat detection and alerting
```
Example: Continuous Monitoring
10:00-10:05 → Window 1 (5 threats detected)
10:01-10:06 → Window 2 (7 threats detected) ← Alert!
10:02-10:07 → Window 3 (8 threats detected) ← Alert!
10:03-10:08 → Window 4 (3 threats detected)

Result: Immediate detection of threat spike at 10:01
```

**Table**: `threat_summary_sliding_5min`

**Metrics**:
- Threat rate (threats per minute)
- Trend detection (increasing/decreasing)
- Anomaly detection (sudden spikes)
- Real-time dashboards

### 3. Tumbling Windows (Compliance Reporting)

**Purpose**: Fixed, non-overlapping time periods for reporting

**Configurations**: 5, 15, 30, 60 minutes

**Use Case**: Compliance and operational reporting

#### 5-Minute Windows
```
Use Case: Operational Monitoring
- Real-time operations dashboard
- Immediate incident response
- System health checks

Example:
10:00-10:05 → 150 events, 12 threats (8%)
10:05-10:10 → 145 events, 8 threats (5.5%)
10:10-10:15 → 160 events, 15 threats (9.4%)
```

#### 15-Minute Windows
```
Use Case: Tactical Analysis
- Shift-level reporting
- Incident investigation
- Pattern identification

Example:
10:00-10:15 → 455 events, 35 threats (7.7%)
10:15-10:30 → 470 events, 28 threats (6.0%)
10:30-10:45 → 445 events, 42 threats (9.4%)
```

#### 30-Minute Windows
```
Use Case: Hourly Security Briefings
- Management reporting
- Trend analysis
- Resource allocation

Example:
10:00-10:30 → 925 events, 63 threats (6.8%)
10:30-11:00 → 915 events, 70 threats (7.7%)
```

#### 60-Minute Windows
```
Use Case: Daily Reports and Compliance
- Executive summaries
- Compliance documentation (SOC 2, PCI-DSS)
- Historical trending

Example:
10:00-11:00 → 1,840 events, 133 threats (7.2%)
11:00-12:00 → 1,920 events, 145 threats (7.6%)
```

**Tables**: 
- `threat_summary_tumbling_5min`
- `threat_summary_tumbling_15min`
- `threat_summary_tumbling_30min`
- `threat_summary_tumbling_60min`

## Aggregation Metrics

Each aggregation table includes the following metrics:

### Window Information
- `window_start` - Window start timestamp
- `window_end` - Window end timestamp
- `window_duration_minutes` - Window size (for tumbling/sliding)
- `session_duration_seconds` - Session length (for session windows)

### Grouping Dimensions
- `customer_id` - Customer identifier
- `device_id` - Device identifier
- `device_name` - Device name

### Event Counts
- `total_events` - Total events in window
- `threat_count` - Number of threats detected
- `threat_percentage` - Percentage of events that are threats

### Severity Breakdown
- `critical_severity_count` - Critical threats
- `high_severity_count` - High severity threats
- `medium_severity_count` - Medium severity threats
- `low_severity_count` - Low severity threats

### Threat Analysis
- `unique_threat_types` - Number of distinct threat types
- `malware_count` - Malware detections
- `data_exfiltration_count` - Data exfiltration attempts
- `failed_login_count` - Failed authentication attempts

### Action Results
- `blocked_count` - Actions blocked by security policy
- `allowed_count` - Actions allowed
- `event_burst_detected` - Rapid event succession (session windows only)

## Query Examples

### Real-Time Threat Monitoring

```sql
-- Current threat levels (last 5 minutes)
SELECT 
  device_name,
  threat_count,
  threat_percentage,
  critical_severity_count,
  high_severity_count,
  window_start,
  window_end
FROM data_generator.threat_summary_sliding_5min
WHERE window_end >= CURRENT_TIMESTAMP - INTERVAL '10' MINUTE
ORDER BY threat_percentage DESC
LIMIT 20;
```

### Attack Campaign Investigation

```sql
-- Identify attack campaigns (session windows)
SELECT 
  device_name,
  window_start,
  window_end,
  session_duration_seconds,
  total_events,
  threat_count,
  event_burst_detected,
  unique_threat_types
FROM data_generator.threat_sessions_5min_gap
WHERE threat_count > 10
  AND event_burst_detected = true
ORDER BY session_duration_seconds DESC;
```

### Compliance Reporting

```sql
-- Hourly security summary for compliance
SELECT 
  DATE_TRUNC('hour', window_start) as report_hour,
  COUNT(DISTINCT device_id) as devices_monitored,
  SUM(total_events) as total_events,
  SUM(threat_count) as total_threats,
  AVG(threat_percentage) as avg_threat_percentage,
  SUM(critical_severity_count) as critical_threats,
  SUM(blocked_count) as threats_blocked
FROM data_generator.threat_summary_tumbling_60min
WHERE window_start >= CURRENT_DATE - INTERVAL '7' DAY
GROUP BY DATE_TRUNC('hour', window_start)
ORDER BY report_hour DESC;
```

### Device Risk Scoring

```sql
-- Identify high-risk devices
SELECT 
  device_name,
  COUNT(*) as window_count,
  AVG(threat_percentage) as avg_threat_rate,
  MAX(threat_count) as max_threats_in_window,
  SUM(critical_severity_count) as total_critical_threats,
  SUM(data_exfiltration_count) as data_exfil_attempts
FROM data_generator.threat_summary_tumbling_15min
WHERE window_start >= CURRENT_TIMESTAMP - INTERVAL '24' HOUR
GROUP BY device_name
HAVING AVG(threat_percentage) > 5.0
ORDER BY avg_threat_rate DESC;
```

### Trend Analysis

```sql
-- Threat trends over time
SELECT 
  DATE_TRUNC('day', window_start) as day,
  COUNT(*) as windows,
  AVG(threat_percentage) as avg_threat_rate,
  SUM(malware_count) as malware_detections,
  SUM(data_exfiltration_count) as data_exfil_attempts,
  SUM(failed_login_count) as failed_logins
FROM data_generator.threat_summary_tumbling_60min
WHERE window_start >= CURRENT_DATE - INTERVAL '30' DAY
GROUP BY DATE_TRUNC('day', window_start)
ORDER BY day;
```

## Security Operations Workflows

### 1. Real-Time Monitoring (SOC Analyst)

**Objective**: Detect and respond to threats as they occur

**Workflow**:
1. Monitor sliding window table for threat spikes
2. Alert when threat_percentage > threshold (e.g., 10%)
3. Investigate affected devices
4. Initiate incident response

**Query**: Sliding window table, updated every minute

### 2. Incident Investigation (Security Engineer)

**Objective**: Understand attack timeline and scope

**Workflow**:
1. Identify suspicious session in session window table
2. Query raw events table for detailed event sequence
3. Analyze attack pattern and techniques
4. Document findings for remediation

**Query**: Session window + raw events tables

### 3. Compliance Reporting (Compliance Officer)

**Objective**: Generate reports for auditors

**Workflow**:
1. Query tumbling window tables for fixed time periods
2. Aggregate metrics by day/week/month
3. Generate charts and summaries
4. Export to compliance documentation

**Query**: Tumbling window tables (15min, 30min, 60min)

### 4. Threat Hunting (Threat Hunter)

**Objective**: Proactively search for hidden threats

**Workflow**:
1. Analyze historical patterns in tumbling windows
2. Identify anomalies and outliers
3. Deep-dive into raw events for suspicious activity
4. Create detection rules for future threats

**Query**: All tables, historical data via Iceberg time travel

## Performance Characteristics

### Throughput
- **Events per second**: 1,000 - 10,000 (configurable via parallelism)
- **Latency**: < 5 seconds (event time to Iceberg write)
- **Checkpoint interval**: 60 seconds

### Scalability
- **Horizontal scaling**: Increase parallelism and KPUs
- **Kafka partitions**: Match Flink parallelism for optimal throughput
- **Iceberg writes**: Batched for efficiency

### Resource Usage
- **Base configuration**: 1 KPU (~$0.11/hour)
- **Recommended**: 2-4 KPUs for production workloads
- **Storage**: ~1-5 GB/day per 1M events (compressed Parquet)
